## 1. 引言

### 1.1 项目背景与意义

随着移动支付的全面普及，个人日常消费行为呈现出高频化、碎片化的特征。微信支付作为国内主流移动支付工具，用户在日常生活中频繁产生各类消费记录，然而绝大多数用户缺乏持续、系统性的记账习惯。研究表明，个人财务管理意识的缺失是导致"月光族"现象和消费失控的重要诱因之一，而传统的记账工具因手动录入繁琐、操作路径长等问题，用户留存率普遍偏低。

在此背景下，本项目拟开发一款基于微信小程序生态的个人记账与AI理财助手——"昕辰记账"。该应用的核心创新点在于利用微信小程序原生提供的消息转发能力，允许用户将微信支付公众号推送的支付通知消息一键转发至小程序，由程序自动完成金额提取、商户识别与分类归因，将传统记账的录入环节压缩至近乎为零。同时，引入人工智能技术，基于用户的收支数据自动生成消费画像和钱包管理建议，帮助用户从"数据记录"上升到"行为洞察"。

本系统在设计理念上贯彻"隐私优先"原则：所有记账数据百分之百存储于用户设备本地（wx.Storage），核心记账功能无需网络连接即可运行；AI分析模块采用"本地规则引擎 + 可选云端大模型"的双轨架构，仅在用户主动配置API密钥且明确授权后，才将脱敏聚合数据发送至第三方AI服务商，从根本上保障用户个人隐私安全。

本项目的开发与推广，对于提升个人财务管理效率、培养理性消费习惯、推动金融科技普惠化具有积极的实践意义。

### 1.2 文档范围

本文档为"昕辰记账"微信小程序的软件需求分析说明书，全面描述系统的功能性需求、非功能性需求及数据需求。文档受众包括开发团队、测试人员、UI/UX设计师及项目管理人员，旨在为后续的系统设计、编码实现和测试验证提供明确的需求基准。

## 2. 功能性需求

### 2.1 微信支付消息解析模块

#### 2.1.1 模块概述

本模块为系统的核心差异化功能，负责接收用户转发的微信支付通知消息，自动解析交易金额、交易方向、商户信息，并匹配预设分类规则实现智能归类。

#### 2.1.2 用例描述

**用例 UC-01：转发消息接收与识别**

| 项目 | 说明 |
|:---|:---|
| **用例编号** | UC-01 |
| **用例名称** | 转发消息接收与识别 |
| **参与者** | 用户（主要参与者）、微信客户端（次要参与者） |
| **前置条件** | 用户已安装并授权"昕辰记账"小程序；微信聊天中存在微信支付通知消息 |
| **触发条件** | 用户长按微信支付通知消息，依次点击"更多打开方式"→"昕辰记账" |
| **基本流程** | 1. 小程序启动，在App.onLaunch/App.onShow中检测referrerInfo.extraData<br>2. 识别当前启动场景为转发消息进入（非普通启动）<br>3. 提取extraData中的消息文本内容<br>4. 将文本传入消息解析引擎<br>5. 若解析成功，弹出确认半弹层展示解析结果；若失败，记录解析日志并提示用户 |
| **后置条件** | 解析结果展示于确认弹层，等待用户确认或修改 |
| **异常流程** | 若extraData为空或非文本格式，则判定为普通启动，进入首页 |
| **备注** | 场景值判断在App.onLaunch/App.onShow中实现 |

**用例 UC-02：消息文本解析**

| 项目 | 说明 |
|:---|:---|
| **用例编号** | UC-02 |
| **用例名称** | 微信支付通知文本解析 |
| **参与者** | 系统（自动执行） |
| **前置条件** | 已获取消息文本内容 |
| **触发条件** | UC-01流程中调用解析引擎 |
| **基本流程** | 1. 使用正则表达式匹配交易金额（支持¥符号及多种金额格式）<br>2. 通过关键词匹配判断交易方向：含"消费""支付"→支出；含"收款""转入""退款""到账""退回""红包"→收入<br>3. 提取商户名称或交易描述文本<br>4. 将商户名/描述与内置关键词映射表进行匹配，自动归入对应分类<br>5. 返回结构化解析结果 |
| **后置条件** | 返回包含金额、类型、分类、商户描述的解析结果对象；若正则未匹配到金额则返回失败状态 |
| **异常流程** | 若正则未匹配到金额，返回status="failed"，failReason记录具体原因 |
| **备注** | 特别注意退款场景：退款通知关键词（退款、到账、退回）应判定为收入；解析需在500ms内完成 |

## 4. 数据需求

### 4.1 数据存储方案概述

本系统基于微信小程序wx.Storage API实现纯客户端数据持久化，不依赖远程数据库。数据以JSON格式序列化存储，各数据实体对应独立的Storage Key。由于wx.Storage的单Key上限为1MB、总容量上限为10MB，需在设置页面提示用户定期导出CSV备份。

### 4.2 核心数据结构定义

#### 4.2.1 交易记录表（transactions）

> **Storage Key**: `transactions`  
> **数据类型**: `Array<Transaction>`  
> **容量估算**: 单条约200字节，10MB上限下可存储约50,000条记录

| 字段名 | 数据类型 | 必填 | 说明 |
|:---|:---|:---|:---|
| `id` | String | 是 | 唯一标识符，采用时间戳+随机数生成，如`"tx_1751342000000_a3f2"` |
| `amount` | Number | 是 | 交易金额，单位元，保留两位小数，如`38.00` |
| `type` | String | 是 | 交易类型，枚举值：`"income"`（收入）/ `"expense"`（支出） |
| `categoryId` | String | 是 | 关联分类ID，外键指向`categories`表中的`id`字段 |
| `categoryName` | String | 是 | 分类名称（冗余字段，提升列表渲染效率） |
| `accountId` | String | 是 | 关联账户ID，外键指向`accounts`表中的`id`字段 |
| `accountName` | String | 是 | 账户名称（冗余字段） |
| `description` | String | 是 | 商户名称或交易描述，如`"美团外卖"` |
| `note` | String | 否 | 用户手动填写的备注文字，可为`null` |
| `source` | String | 是 | 数据来源，枚举值：`"wechat"`（微信转发）/ `"manual"`（手动录入） |
| `timestamp` | Number | 是 | 实际交易时间戳（毫秒），支持补录历史交易 |
| `createdAt` | Number | 是 | 记录创建时间戳（毫秒），系统自动生成 |
| `aiTag` | String | 否 | AI智能标签，枚举值：`"必要支出"` / `"可选支出"` / `"冲动消费"` / `"固定开销"`，可为`null` |

#### 4.2.2 分类表（categories）

> **Storage Key**: `categories`  
> **数据类型**: `Array<Category>`

| 字段名 | 数据类型 | 必填 | 说明 |
|:---|:---|:---|:---|
| `id` | String | 是 | 唯一标识符，如`"cat_001"` |
| `name` | String | 是 | 分类名称，如`"餐饮"` |
| `emoji` | String | 是 | 分类图标Emoji，如`"🍔"` |
| `isPreset` | Boolean | 是 | 是否为系统预置分类，预置分类不可删除 |
| `sortOrder` | Number | 是 | 排序序号，用于控制列表展示顺序 |

**预置数据**（共10条）：

| id | name | emoji | isPreset | sortOrder |

### 4.4 Storage容量估算与约束

| 参数 | 数值 |
|:---|:---|
| 单Key存储上限 | 1 MB |
| 总存储上限 | 10 MB |
| 单条Transaction记录估算大小 | 约200字节 |
| 理论最大记录数（10MB / 200字节） | 约50,000条 |
| 单条AiReport记录估算大小 | 约2,000字节（含Markdown内容） |
| 建议AiReport保留条数 | ≤500条（约1MB） |
| 建议parse_logs保留条数 | ≤200条（约100KB） |

**Storage超限应对策略**：

1. 设置页面醒目提示当前Storage使用量和剩余空间
2. 当总使用量超过8MB（80%）时，主动提示用户导出CSV备份
3. parse_logs采用FIFO策略，最多保留200条，超出后自动清理最旧记录
4. AI建议历史（ai_reports）列表支持用户手动清理

### 实现与需求差异说明

| 需求项 | 原需求 | 实际实现 | 原因 |
|:---|:---|:---|:---|
| **1. 支付消息入口** | 转发消息→更多打开方式→小程序 | 复制消息→打开小程序→粘贴识别 | 微信支付消息不支持「更多打开方式」 |
| **2. 统计图表** | ECharts 饼图+折线图 | Canvas 2D 环形图 + 纯 View 柱状图 | 无需~900KB ECharts 依赖；Canvas 折线图原生组件浮层问题改为 View 柱状图解决 |
| 2.3 保存动效 | 勾选动画500ms + 金额数字滚动800ms | 简化为保存后关闭弹层+列表刷新 | 功能优先，动效后续打磨 |
| 10 设置页 | 原计划P4实现 | P1即完成设置页框架 | 用户需要分类/账户管理入口 |
| 2.5 左滑删除 | 简单左滑+静态按钮 | QQ风格跟手拖拽+弹性吸边 | 用户体验优化 |
| 保存按钮颜色 | 樱花粉色(#F8BBD0) | 薄荷绿(#A5D6A7) | 用户反馈调整 |
| 编辑页金额输入 | `<input type="digit">` | 自定义数字键盘 | 与记账体验保持一致 |
| **3. CSV 导入** | 逐行解析后直接入库 | 导入时自动匹配分类名/账户名（不区分大小写+模糊匹配） | 避免导入后全部分类显示为「其他」 |
| **4. AI 历史** | 保存AI对话到Storage+历史页面回溯 | 纯内存存储，切换Tab不丢但退出清空；历史页移除 | 简化复杂度，减少Storage占用 |
